# Design Documentation

## Overview

s3-pypi-server implements a PEP 503 compliant Python package repository using AWS managed services. The design prioritizes simplicity, low operational overhead, and cost efficiency. The core read path uses API Gateway's native S3 integration (no Lambda). An optional Lambda authorizer adds authentication when security is required.

## Architecture

```
┌──────────┐     ┌──────────────┐     ┌─────────────┐     ┌───────────┐
│   pip    │────▶│  CloudFront  │────▶│ API Gateway │────▶│  S3       │
│  client  │     │  (CDN/TLS)   │     │  (REST API) │     │  (store)  │
└──────────┘     └──────────────┘     └─────────────┘     └───────────┘
                                             │
                                       ┌─────┴─────┐
                                       │ Authorizer │ (optional)
                                       │  Lambda    │
                                       └─────┬─────┘
                                             │
                                  ┌──────────┼──────────┐
                                  ▼                     ▼
                             DynamoDB              Secrets Mgr
                            (API keys)           (LDAP config)

┌──────────┐                                               ┌───────────┐
│  s3pypi  │──────────────────────────────────────────────▶│  S3       │
│   CLI    │──────────────────────────────────────────────▶│CloudFront │
└──────────┘          upload + invalidate                  └───────────┘
     │
     ├──▶ DynamoDB (API key CRUD)
     └──▶ Secrets Manager (LDAP config)
```

### Components

**S3 Bucket** — Stores two types of objects:
- `packages/{normalized_name}/{filename}` — Distribution files (wheels, sdists).
- `simple/{normalized_name}/index.html` — PEP 503 detail pages listing files for a package.
- `simple/index.html` — PEP 503 root index listing all packages.

The bucket name is randomly generated by CloudFormation (no explicit name). Retrieve it from the `BucketName` stack output.

Encryption (AES-256 or KMS), versioning, and public access blocking are enabled by default.

**API Gateway** — REST API with AWS service integration (no Lambda for reads). Maps:
- `GET /simple/` → `s3://bucket/simple/index.html`
- `GET /simple/{package}/` → `s3://bucket/simple/{package}/index.html`
- `GET /simple/{package}/{file}` → `s3://bucket/packages/{package}/{file}`

Returns `text/html` for index pages and binary passthrough for distribution files.

**CloudFront** — CDN layer providing:
- HTTPS termination with TLS 1.2 minimum.
- Caching with configurable TTL (default 300s).
- Origin pointed at the API Gateway `prod` stage.
- Optional custom domain with ACM certificate.

**Lambda Authorizer** (optional) — Request authorizer that validates:
1. Bearer tokens against a DynamoDB API key table.
2. Basic Auth credentials against an LDAP/AD server.

**DynamoDB Table** (optional) — Stores API keys with partition key `api_key`. The table name is randomly generated by CloudFormation. Retrieve it from the `APIKeyTableName` stack output. Used by the authorizer and managed via the CLI.

**Secrets Manager** (optional) — Stores LDAP connection configuration (host, bind_user, bind_password, entitlement_group). Created by CloudFormation, populated via CLI.

**KMS Key** (optional) — Customer Managed Key that encrypts S3, DynamoDB, and CloudWatch Logs at rest.

**CLI (`s3pypi`)** — Command-line tool that:
1. Deploys the CloudFormation stack and auto-saves outputs to config.
2. Uploads distribution files and regenerates index pages.
3. Manages API keys (create, list, get, delete, update).
4. Configures LDAP secrets.
5. Optionally creates CloudFront invalidations.

## Key Design Decisions

### No Lambda for reads
API Gateway's native S3 integration handles all read operations. This eliminates cold starts, reduces cost, and simplifies the deployment. The only Lambda is the optional authorizer.

### Index regeneration on upload
Rather than generating index pages on-the-fly per request, the CLI regenerates static HTML files in S3 after each upload. This means reads are a single S3 GetObject — fast and cheap.

### PEP 503 compliance
All generated HTML follows PEP 503 (Simple Repository API) including:
- `<!DOCTYPE html>` declaration
- `<meta name="pypi:repository-version" content="1.0">` tag
- Normalized package names in URLs (lowercase, hyphens)

### Package name normalization
Follows PEP 503: lowercase, replace runs of `[-_.]` with a single hyphen. This is applied consistently in S3 keys, HTML generation, and URL routing.

### Dual authentication
The authorizer supports two methods to cover different use cases:
- **`__token__` Basic Auth** — For CI/CD pipelines and automated systems. The username is always `__token__` and the password is the API key (matching PyPI conventions). Keys have an `access` attribute: `read` (GET only) or `read/write` (GET + POST).
- **LDAP/AD Basic Auth** — For developers using corporate credentials. The authorizer binds to the LDAP server and checks group membership. Members of `entitlement_group` get read access; members of `write_entitlement_group` get read/write access.

The `__token__` username is checked first. If the username is anything else, LDAP authentication is attempted.

### Twine-compatible uploads
When the authorizer is enabled, a POST method on `/simple/` is backed by an Upload Lambda. This Lambda parses multipart/form-data (twine's upload format), extracts the distribution file, uploads it to S3, and regenerates index pages. Only users/keys with write access can POST.

### Optional KMS encryption
All encryption is opt-in via `EnableKMSEncryption=true`. When enabled, a single CMK encrypts:
- S3 bucket (server-side encryption)
- DynamoDB table (SSE)
- CloudWatch Log Groups

The key policy grants the account root full access, allowing IAM policies to control key usage.

### Optional VPC placement
When `VpcId` and `SubnetIds` are provided, the authorizer Lambda is placed inside the VPC. This allows it to reach LDAP/AD servers on private networks. A Security Group with egress-only rules is created automatically.

### 30-day log retention
All Lambda functions and the API Gateway stage have dedicated CloudWatch Log Groups with 30-day retention. This provides operational visibility without unbounded log accumulation.

## Data Flow: Upload

1. User runs `s3pypi upload my_pkg-1.0.0.whl --bucket my-bucket`.
2. CLI parses filename → `(my_pkg, 1.0.0, .whl)`.
3. Normalizes name → `my-pkg`.
4. Uploads file to `s3://my-bucket/packages/my-pkg/my_pkg-1.0.0.whl`.
5. Lists all files under `packages/my-pkg/` and writes detail page to `simple/my-pkg/index.html`.
6. Lists all package prefixes under `packages/` and writes root index to `simple/index.html`.
7. If `--cloudfront-distribution-id` is provided, invalidates `/simple/` and `/simple/my-pkg/`.

## Data Flow: Install (with authorizer)

1. `pip` requests `GET /simple/` with `Authorization: Bearer <key>` header.
2. API Gateway invokes the authorizer Lambda.
3. Authorizer looks up the key in DynamoDB → Allow policy.
4. API Gateway fetches `simple/index.html` from S3 → returns to pip.
5. `pip` finds the package link and requests `GET /simple/my-pkg/` → detail page.
6. `pip` finds the file link and requests `GET /simple/my-pkg/my_pkg-1.0.0.whl` → binary download.

## Data Flow: Install (without authorizer)

1. `pip` requests `GET /simple/` → CloudFront → API Gateway → S3 `simple/index.html`.
2. `pip` finds the package link and requests `GET /simple/my-pkg/` → detail page.
3. `pip` finds the file link and requests `GET /simple/my-pkg/my_pkg-1.0.0.whl` → binary download from `packages/my-pkg/my_pkg-1.0.0.whl`.

## Authorizer Authentication Flow

```
Request arrives with Authorization header
    │
    └── Starts with "Basic " ?
            │
            ├── Yes → Decode base64 → username:password
            │           │
            │           ├── username == "__token__" ?
            │           │       │
            │           │       ├── Yes → DynamoDB GetItem(api_key=password)
            │           │       │           ├── Item found, access=read → ALLOW (GET only)
            │           │       │           ├── Item found, access=read/write → ALLOW (GET + POST)
            │           │       │           └── Not found → DENY
            │           │       │
            │           │       └── No → LDAP/AD authentication
            │           │               ├── Bind succeeds + member of write_entitlement_group → ALLOW (GET + POST)
            │           │               ├── Bind succeeds + member of entitlement_group → ALLOW (GET only)
            │           │               └── Bind fails or not in any group → DENY
            │           │
            │           └── Decode fails → DENY
            │
            └── No → DENY
```
