# Requirements Document

## Introduction

This document defines the requirements for a private PyPI server infrastructure deployed on AWS. The system uses an S3 bucket as the package repository backend, API Gateway as the HTTP front-end implementing the PyPI Simple Repository API, and CloudFront as a CDN cache layer. The entire infrastructure is defined as CloudFormation and deployed via the AWS CLI. A Python CLI tool is provided for uploading Python packages to the S3 bucket, managing API keys, and configuring LDAP secrets.

The system supports optional security hardening through KMS encryption at rest for all resources, an API Gateway authorizer with dual authentication methods (LDAP/AD and DynamoDB API keys), and optional VPC placement for Lambda functions. Authentication uses the `__token__` username convention for API key-based access (matching PyPI standards), and supports role-based access control with separate read and write permissions. LDAP/AD authorization supports two entitlement groups: one for read access and one for write (upload) access. The API Gateway supports package uploads via POST (twine-compatible), enabling standard Python packaging workflows. All Lambda functions and API Gateways are configured with CloudWatch LogGroups with 30-day retention policies for operational visibility.

The project follows the MIT license, validates with pylint and bandit, and includes comprehensive documentation and unit tests.

## Glossary

- **S3_Bucket**: An Amazon S3 bucket that stores Python package distribution files and the PyPI Simple Repository API index pages. The bucket name is randomly generated by CloudFormation.
- **API_Gateway**: An Amazon API Gateway (REST API) that serves HTTP requests for the PyPI Simple Repository API by proxying to the S3_Bucket.
- **CloudFront_Distribution**: An Amazon CloudFront distribution that caches responses from the API_Gateway to reduce latency and load.
- **CloudFormation_Stack**: An AWS CloudFormation stack defined in a YAML template that provisions all AWS resources (S3_Bucket, API_Gateway, CloudFront_Distribution, IAM roles, and related resources).
- **Upload_CLI**: A Python command-line tool that uploads Python package distribution files (.tar.gz, .whl) to the S3_Bucket and regenerates the Simple Repository API index pages.
- **Simple_Repository_API**: The PyPI Simple Repository API (PEP 503) that provides a standardized interface for pip to discover and download packages.
- **Package_Index_Page**: An HTML page listing all available packages, served at the repository root path.
- **Package_Detail_Page**: An HTML page listing all distribution files for a specific package, served at the package name path.
- **Distribution_File**: A Python package archive file in .tar.gz (sdist) or .whl (bdist_wheel) format.
- **Stack_Deployer**: The `deploy` subcommand of the Upload_CLI that uses boto3 CloudFormation APIs to create or update the CloudFormation_Stack, accepting parameters and recording outputs to the Configuration_File.
- **Configuration_File**: A JSON file at `~/.s3pypi/config.json` that stores default CLI settings (bucket name, CloudFront distribution ID, DynamoDB table name, Secrets Manager secret ARN).
- **ACM_Certificate**: An AWS Certificate Manager certificate used to enable HTTPS with a custom domain on the CloudFront_Distribution.
- **KMS_Key**: An AWS KMS Customer Managed Key used to encrypt resources at rest when KMS encryption is enabled.
- **Authorizer**: An API Gateway Lambda authorizer that validates requests before they reach the backend integration.
- **LDAP_Secret**: An AWS Secrets Manager secret containing LDAP/AD connection configuration (host, bind user, bind password, read entitlement group, and write entitlement group).
- **API_Key_Table**: A DynamoDB table that stores API keys used for authenticating requests to the API Gateway with a partition key of `api_key` and an `access` attribute indicating `read` or `read/write` permissions. The table name is randomly generated by CloudFormation.
- **Upload_Lambda**: A Lambda function that handles package uploads via POST requests to the API Gateway, enabling twine-compatible publishing.
- **Authorizer_Lambda**: A Lambda function that implements the API Gateway authorizer logic, supporting both LDAP/AD and API key authentication methods.
- **Security_Group**: A VPC Security Group assigned to all Lambda functions when VPC support is enabled.
- **Lambda_LogGroup**: A CloudWatch Logs LogGroup associated with a Lambda function, configured with a 30-day retention policy.
- **API_Gateway_LogGroup**: A CloudWatch Logs LogGroup used for API Gateway access logging, configured with a 30-day retention policy.
- **CLI_API_Key_Manager**: A CLI subcommand that performs CRUD operations on API keys stored in the API_Key_Table.
- **CLI_Secret_Manager**: A CLI subcommand that sets or updates the LDAP configuration stored in the LDAP_Secret.

## Requirements

### Requirement 1: S3 Bucket Package Storage

**User Story:** As a DevOps engineer, I want Python packages stored in an S3 bucket, so that I have durable, scalable storage for my private package repository.

#### Acceptance Criteria

1. THE CloudFormation_Stack SHALL provision an S3_Bucket with a randomly generated unique bucket name (CloudFormation auto-naming).
2. THE CloudFormation_Stack SHALL configure the S3_Bucket with server-side encryption enabled using AES-256.
3. THE CloudFormation_Stack SHALL configure the S3_Bucket to block all public access.
4. THE CloudFormation_Stack SHALL configure the S3_Bucket with versioning enabled.

### Requirement 2: API Gateway PyPI Interface

**User Story:** As a developer, I want an API Gateway that implements the PyPI Simple Repository API, so that pip can discover and install packages from my private repository.

#### Acceptance Criteria

1. THE CloudFormation_Stack SHALL provision an API_Gateway REST API that proxies GET requests to the S3_Bucket.
2. WHEN a GET request is received at the root path, THE API_Gateway SHALL return the Package_Index_Page from the S3_Bucket.
3. WHEN a GET request is received at a package name path, THE API_Gateway SHALL return the Package_Detail_Page for that package from the S3_Bucket.
4. WHEN a GET request is received for a Distribution_File path, THE API_Gateway SHALL return the binary Distribution_File from the S3_Bucket with the correct content type.
5. IF a requested resource does not exist in the S3_Bucket, THEN THE API_Gateway SHALL return an HTTP 404 response.
6. THE CloudFormation_Stack SHALL provision an IAM role that grants the API_Gateway read-only access to the S3_Bucket.
7. THE API_Gateway SHALL set the Content-Type response header to "text/html" for Package_Index_Page and Package_Detail_Page responses.

### Requirement 3: CloudFront Caching Layer

**User Story:** As a DevOps engineer, I want CloudFront to cache responses from the API Gateway, so that package downloads are fast and API Gateway load is reduced.

#### Acceptance Criteria

1. THE CloudFormation_Stack SHALL provision a CloudFront_Distribution with the API_Gateway as its origin.
2. THE CloudFront_Distribution SHALL cache responses from the API_Gateway with a default TTL of 300 seconds.
3. THE CloudFront_Distribution SHALL forward the Host header to the API_Gateway origin.
4. WHEN a cached resource is updated in the S3_Bucket, THE Upload_CLI SHALL create a CloudFront invalidation for the affected Package_Index_Page and Package_Detail_Page paths.
5. THE CloudFront_Distribution SHALL use the default CloudFront domain name (no custom domain required).
6. THE CloudFormation_Stack SHALL configure the CloudFront_Distribution to use TLS 1.2 as the minimum protocol version.

### Requirement 12: Custom Domain and ACM Certificate

**User Story:** As a DevOps engineer, I want to optionally use a custom domain name (e.g., `pypi.example.org`) with an ACM certificate for my private PyPI server, so that my team can access it via a memorable, branded URL.

#### Acceptance Criteria

1. THE CloudFormation_Stack SHALL accept an optional `DomainName` parameter for a custom CloudFront CNAME (e.g., `pypi.example.org`).
2. THE CloudFormation_Stack SHALL accept an optional `AcmCertificateArn` parameter for an AWS Certificate Manager certificate ARN.
3. WHEN both `DomainName` and `AcmCertificateArn` are provided, THE CloudFront_Distribution SHALL be configured with the domain as an alias and the ACM certificate for TLS.
4. WHEN `DomainName` or `AcmCertificateArn` are not provided, THE CloudFront_Distribution SHALL use the default CloudFront domain name and default certificate (existing behavior).
5. THE CloudFormation_Stack SHALL use CloudFormation conditions to conditionally apply the custom domain and certificate configuration.
6. THE CloudFormation_Stack SHALL output the effective endpoint domain — the custom `DomainName` if provided, otherwise the default CloudFront domain.

### Requirement 4: CloudFormation Deployment

**User Story:** As a DevOps engineer, I want to deploy the entire infrastructure from the Python CLI with a single command, so that I can reproducibly provision the environment without external scripts.

#### Acceptance Criteria

1. THE CloudFormation_Stack SHALL define all AWS resources (S3_Bucket, API_Gateway, CloudFront_Distribution, IAM roles) in a single YAML template file bundled inside the Python package.
2. THE CloudFormation_Stack SHALL export the CloudFront_Distribution domain name as a stack output named "PyPIEndpoint".
3. THE CloudFormation_Stack SHALL export the S3_Bucket name as a stack output named "BucketName".
4. THE Upload_CLI SHALL provide a `deploy` subcommand that deploys the CloudFormation stack using boto3.
5. THE `deploy` subcommand SHALL require a `--stack-name` argument specifying the CloudFormation stack name.
6. THE `deploy` subcommand SHALL accept optional `--profile` argument matching a boto3/AWS CLI named profile.
7. THE `deploy` subcommand SHALL accept optional `--region` argument defaulting to `us-east-1`.
8. THE `deploy` subcommand SHALL accept optional CloudFormation parameter overrides as positional arguments in `Key=Value` format.
9. WHEN the deployment succeeds, THE `deploy` subcommand SHALL automatically record the stack outputs (BucketName, PyPIEndpoint, APIKeyTableName, LDAPSecretArn) into the Configuration_File.
10. IF the CloudFormation deployment fails, THEN THE `deploy` subcommand SHALL report the failure reason from CloudFormation events to stderr and exit with a non-zero exit code.
11. THE CloudFormation template SHALL be included as package data inside the `s3pypi` Python package so it is available at runtime without external files.
12. THE CloudFormation_Stack SHALL use parameters for configurable values including stack name prefix and cache TTL.

### Requirement 5: Python Upload CLI

**User Story:** As a developer, I want a Python CLI tool to upload packages to the S3 bucket, so that I can publish private packages without manual S3 operations.

#### Acceptance Criteria

1. THE Upload_CLI SHALL accept a Distribution_File path as a required positional argument.
2. THE Upload_CLI SHALL accept a bucket name as a required argument via the `--bucket` flag.
3. WHEN a Distribution_File is provided, THE Upload_CLI SHALL upload the file to the S3_Bucket under the path `packages/{package_name}/{filename}`.
4. WHEN a Distribution_File is uploaded, THE Upload_CLI SHALL regenerate the Package_Detail_Page for the affected package by listing all Distribution_Files for that package in the S3_Bucket.
5. WHEN a Distribution_File is uploaded, THE Upload_CLI SHALL regenerate the Package_Index_Page by listing all packages in the S3_Bucket.
6. THE Upload_CLI SHALL generate Package_Index_Page and Package_Detail_Page HTML that conforms to PEP 503 Simple Repository API format.
7. IF the specified Distribution_File does not exist on the local filesystem, THEN THE Upload_CLI SHALL exit with a non-zero exit code and print an error message to stderr.
8. IF the S3 upload operation fails, THEN THE Upload_CLI SHALL exit with a non-zero exit code and print the error details to stderr.
9. THE Upload_CLI SHALL extract the package name from the Distribution_File filename using standard Python packaging naming conventions.
10. THE Upload_CLI SHALL accept an optional `--cloudfront-distribution-id` flag and, WHEN provided, create a CloudFront invalidation for the updated index pages.
11. WHEN `--bucket` or `--cloudfront-distribution-id` are not provided on the `upload` subcommand, THE Upload_CLI SHALL fall back to values stored in the configuration file if one exists.

### Requirement 11: CLI Configuration

**User Story:** As a developer, I want to save my default bucket and CloudFront distribution ID so that I don't have to specify them on every upload command.

#### Acceptance Criteria

1. THE Upload_CLI SHALL provide a `configure` subcommand that persists default settings to a configuration file.
2. THE `configure` subcommand SHALL accept `--bucket`, `--cloudfront-distribution-id`, `--api-key-table-name`, and `--ldap-secret-arn` flags and save their values.
3. THE configuration file SHALL be stored at `~/.s3pypi/config.json`.
4. THE configuration file SHALL use JSON format with keys `bucket`, `cloudfront_distribution_id`, `api_key_table_name`, and `ldap_secret_arn`.
5. WHEN the `configure` subcommand is run, THE Upload_CLI SHALL create the `~/.s3pypi/` directory if it does not exist.
6. WHEN the `configure` subcommand is run with a subset of flags, THE Upload_CLI SHALL merge the new values with any existing configuration, preserving values not explicitly overridden.
7. WHEN the `upload` subcommand is run without `--bucket`, THE Upload_CLI SHALL read the bucket from the configuration file. IF neither the flag nor the configuration file provides a bucket, THE Upload_CLI SHALL exit with a non-zero exit code and print an error message to stderr.
8. WHEN the `upload` subcommand is run without `--cloudfront-distribution-id`, THE Upload_CLI SHALL read the distribution ID from the configuration file if present.
9. THE `configure` subcommand SHALL print the saved configuration to stdout after writing.
10. WHEN the `configure` subcommand is run without any flags, THE Upload_CLI SHALL interactively prompt the user for each configuration value (bucket, cloudfront_distribution_id, api_key_table_name, ldap_secret_arn), displaying the current value as the default in brackets, and accepting an empty input to keep the existing value.

### Requirement 13: Optional KMS Encryption

**User Story:** As a security engineer, I want to optionally encrypt all resources at rest with a Customer Managed KMS key, so that I can meet organizational encryption compliance requirements.

#### Acceptance Criteria

1. THE CloudFormation_Stack SHALL accept an optional boolean parameter `EnableKMSEncryption` (default: `false`).
2. WHEN `EnableKMSEncryption` is true, THE CloudFormation_Stack SHALL create a Customer Managed KMS_Key with a key policy granting the account root full access.
3. WHEN `EnableKMSEncryption` is true, THE CloudFormation_Stack SHALL configure the S3_Bucket to use the KMS_Key for server-side encryption instead of AES-256.
4. WHEN `EnableKMSEncryption` is true, THE CloudFormation_Stack SHALL configure the API_Key_Table to use the KMS_Key for encryption at rest.
5. WHEN `EnableKMSEncryption` is true, THE CloudFormation_Stack SHALL configure all Lambda_LogGroups to use the KMS_Key for encryption.
6. WHEN `EnableKMSEncryption` is false, THE CloudFormation_Stack SHALL use AES-256 encryption for the S3_Bucket and AWS-managed encryption for all other resources (existing behavior).
7. THE CloudFormation_Stack SHALL use a CloudFormation condition to conditionally create the KMS_Key and apply KMS encryption configuration.

### Requirement 14: Optional API Gateway Authorizer

**User Story:** As a security engineer, I want to optionally enable an authorizer on the API Gateway, so that only authenticated users and systems can access the private PyPI repository.

#### Acceptance Criteria

1. THE CloudFormation_Stack SHALL accept an optional boolean parameter `EnableAuthorizer` (default: `false`).
2. WHEN `EnableAuthorizer` is true, THE CloudFormation_Stack SHALL create an Authorizer_Lambda that validates incoming requests.
3. WHEN `EnableAuthorizer` is true, THE CloudFormation_Stack SHALL configure the API_Gateway to use the Authorizer_Lambda as a request authorizer on all routes.
4. WHEN `EnableAuthorizer` is false, THE API_Gateway SHALL allow unauthenticated access to all routes (existing behavior).
5. THE CloudFormation_Stack SHALL use a CloudFormation condition to conditionally create the Authorizer and attach the Authorizer to the API_Gateway methods.
6. WHEN `EnableAuthorizer` is true, THE CloudFormation_Stack SHALL create an IAM execution role for the Authorizer_Lambda granting access to the LDAP_Secret, the API_Key_Table, and CloudWatch Logs.

### Requirement 15: LDAP/AD Authentication Method

**User Story:** As a developer, I want to authenticate to the PyPI server using my corporate LDAP/AD credentials, so that I can use existing enterprise identity management for access control.

#### Acceptance Criteria

1. WHEN `EnableAuthorizer` is true, THE CloudFormation_Stack SHALL create an LDAP_Secret in AWS Secrets Manager with an initial empty JSON structure containing keys: `host`, `bind_user`, `bind_password`, `entitlement_group`, and `write_entitlement_group`.
2. WHEN a request includes a Basic Authentication header with username and password, THE Authorizer_Lambda SHALL authenticate the credentials against the LDAP/AD server configured in the LDAP_Secret.
3. WHEN the LDAP/AD server confirms the user credentials are valid and the user is a member of the configured `entitlement_group`, THE Authorizer_Lambda SHALL return an Allow policy granting read access (GET methods).
4. WHEN the LDAP/AD server confirms the user credentials are valid and the user is a member of the configured `write_entitlement_group`, THE Authorizer_Lambda SHALL return an Allow policy granting both read and write access (GET and POST methods).
5. IF the LDAP/AD server rejects the credentials or the user is not a member of either entitlement group, THEN THE Authorizer_Lambda SHALL return a Deny policy.
6. IF the LDAP_Secret cannot be read or the LDAP/AD server is unreachable, THEN THE Authorizer_Lambda SHALL return a Deny policy and log the error to the Lambda_LogGroup.

### Requirement 16: DynamoDB API Key Authentication Method

**User Story:** As a CI/CD pipeline operator, I want to authenticate to the PyPI server using an API key with the `__token__` username convention, so that automated systems can access packages using standard PyPI tooling (pip, twine).

#### Acceptance Criteria

1. WHEN `EnableAuthorizer` is true, THE CloudFormation_Stack SHALL create an API_Key_Table in DynamoDB with a randomly generated table name (CloudFormation auto-naming) and a partition key named `api_key` of type String.
2. WHEN a request includes a Basic Authentication header with username `__token__` and the API key as the password, THE Authorizer_Lambda SHALL look up the password value in the API_Key_Table.
3. WHEN the key value exists in the API_Key_Table and the key's `access` attribute is `read`, THE Authorizer_Lambda SHALL return an Allow policy granting read access (GET methods only).
4. WHEN the key value exists in the API_Key_Table and the key's `access` attribute is `read/write`, THE Authorizer_Lambda SHALL return an Allow policy granting both read and write access (GET and POST methods).
5. IF the key value does not exist in the API_Key_Table, THEN THE Authorizer_Lambda SHALL return a Deny policy.
6. THE Authorizer_Lambda SHALL recognize the `__token__` username in Basic Authentication as the API key authentication method, distinguishing it from LDAP/AD authentication where the username is not `__token__`.
7. THE API_Key_Table SHALL store an `access` attribute for each key with allowed values `read` or `read/write`, defaulting to `read` if not specified.

### Requirement 17: CLI API Key Management

**User Story:** As an administrator, I want a CLI command to manage API keys in the DynamoDB table with configurable access levels, so that I can grant and revoke access for automated systems with appropriate permissions.

#### Acceptance Criteria

1. THE Upload_CLI SHALL provide an `apikey` subcommand with actions: `create`, `list`, `get`, `delete`, and `update`.
2. WHEN the `apikey create` action is invoked, THE CLI_API_Key_Manager SHALL generate a new unique API key, store it in the API_Key_Table with a default `access` value of `read`, and print the key value to stdout.
3. WHEN the `apikey create` action is invoked with an optional `--description` flag, THE CLI_API_Key_Manager SHALL store the description alongside the API key in the API_Key_Table.
4. WHEN the `apikey create` action is invoked with an optional `--access` flag (values: `read` or `read/write`), THE CLI_API_Key_Manager SHALL store the specified access level in the API_Key_Table.
5. WHEN the `apikey list` action is invoked, THE CLI_API_Key_Manager SHALL scan the API_Key_Table and print all API keys with their descriptions and access levels to stdout.
6. WHEN the `apikey get` action is invoked with a key value argument, THE CLI_API_Key_Manager SHALL retrieve and display the API key record including its access level from the API_Key_Table.
7. WHEN the `apikey delete` action is invoked with a key value argument, THE CLI_API_Key_Manager SHALL remove the API key from the API_Key_Table.
8. WHEN the `apikey update` action is invoked with a key value argument and `--access` flag, THE CLI_API_Key_Manager SHALL update the access level of the specified key in the API_Key_Table.
9. IF the specified API key does not exist in the API_Key_Table for `get`, `delete`, or `update` actions, THEN THE CLI_API_Key_Manager SHALL print an error message to stderr and exit with a non-zero exit code.
10. THE `apikey` subcommand SHALL accept a `--table-name` flag to specify the DynamoDB table name, falling back to the Configuration_File if not provided.

### Requirement 18: CLI Secrets Manager Configuration

**User Story:** As an administrator, I want a CLI command to populate the LDAP configuration in Secrets Manager, so that I can configure the authorizer without manual AWS Console operations.

#### Acceptance Criteria

1. THE Upload_CLI SHALL provide a `configure-ldap` subcommand that sets or updates the LDAP configuration in the LDAP_Secret.
2. THE `configure-ldap` subcommand SHALL accept required flags: `--host`, `--bind-user`, `--bind-password`, and `--entitlement-group`.
3. THE `configure-ldap` subcommand SHALL accept an optional `--write-entitlement-group` flag to specify the group that grants write (upload) access.
4. THE `configure-ldap` subcommand SHALL accept a `--secret-arn` flag to specify the Secrets Manager secret ARN, falling back to the Configuration_File if not provided.
5. WHEN the `configure-ldap` subcommand is invoked with all required flags, THE CLI_Secret_Manager SHALL update the LDAP_Secret with the provided values as a JSON object including `host`, `bind_user`, `bind_password`, `entitlement_group`, and `write_entitlement_group`.
6. IF any required flag is missing, THEN THE CLI_Secret_Manager SHALL print an error message to stderr and exit with a non-zero exit code.
7. IF the Secrets Manager update operation fails, THEN THE CLI_Secret_Manager SHALL print the error details to stderr and exit with a non-zero exit code.
8. WHEN the update succeeds, THE CLI_Secret_Manager SHALL print a confirmation message to stdout.

### Requirement 19: Optional VPC Support

**User Story:** As a network engineer, I want to optionally deploy Lambda functions inside a VPC, so that the authorizer can reach LDAP/AD servers on the private network.

#### Acceptance Criteria

1. THE CloudFormation_Stack SHALL accept optional parameters `SubnetIds` (CommaDelimitedList, default: `""`) and `VpcId` (String, default: `""`).
2. WHEN both `SubnetIds` and `VpcId` are provided, THE CloudFormation_Stack SHALL create a Security_Group in the specified VPC with egress-only rules allowing all outbound traffic.
3. WHEN both `SubnetIds` and `VpcId` are provided, THE CloudFormation_Stack SHALL configure all Lambda functions with a VpcConfig specifying the provided SubnetIds and the created Security_Group.
4. WHEN `SubnetIds` or `VpcId` are not provided, THE CloudFormation_Stack SHALL deploy Lambda functions without VPC configuration (existing behavior).
5. THE CloudFormation_Stack SHALL use a CloudFormation condition to conditionally create the Security_Group and apply VPC configuration to Lambda functions.
6. WHEN VPC support is enabled, THE CloudFormation_Stack SHALL add the `ec2:CreateNetworkInterface`, `ec2:DescribeNetworkInterfaces`, and `ec2:DeleteNetworkInterface` permissions to the Lambda execution role.

### Requirement 22: Twine-Compatible Package Upload via API

**User Story:** As a developer, I want to upload packages to the private PyPI server using `twine upload`, so that I can use standard Python packaging workflows without needing the custom CLI for publishing.

#### Acceptance Criteria

1. WHEN `EnableAuthorizer` is true, THE CloudFormation_Stack SHALL create an Upload_Lambda that handles POST requests for package uploads.
2. THE CloudFormation_Stack SHALL add a POST method on the API_Gateway at the `/simple/` resource path, integrated with the Upload_Lambda.
3. THE Upload_Lambda SHALL accept multipart/form-data POST requests containing a distribution file (matching twine's upload format).
4. THE Upload_Lambda SHALL extract the distribution file from the POST body, upload it to the S3_Bucket under `packages/{normalized_name}/{filename}`, and regenerate the affected index pages.
5. THE Upload_Lambda SHALL require write access — the Authorizer_Lambda SHALL only allow POST requests from users/keys with write permissions.
6. THE Upload_Lambda SHALL return HTTP 200 on successful upload, HTTP 400 for malformed requests, and HTTP 500 for internal errors.
7. WHEN authenticating with twine, THE user SHALL use `__token__` as the username and the API key as the password (matching PyPI conventions).
8. THE CloudFormation_Stack SHALL grant the Upload_Lambda IAM permissions to read and write to the S3_Bucket and to create CloudFront invalidations (if a distribution ID is available).
9. THE Upload_Lambda SHALL have a dedicated Lambda_LogGroup with 30-day retention.
10. WHEN `EnableKMSEncryption` is true, THE Upload_Lambda's LogGroup SHALL use the KMS_Key for encryption.
